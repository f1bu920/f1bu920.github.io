面试总结-计算机网络

发表于 2020-10-30 | 分类于 面试

1. OSI七层模型&五层模型

  • 应用层

    为特定应用程序提供数据传输服务，如HTTP、DNS、SMTP等，数据单位为报文

  • 传输层

    为进程提供通用数据传输服务。传输层包括两种协议：传输控制协议TCP，提供面向连接、可靠的数据传输服务，数据单位为报文段；用户数据报协议UDP，提供无连接、不可靠的数据传输服务，数据单位为用户数据报。TCP提供完整性服务，UDP提供及时性服务。

  • 网络层

    网络层负责为分组交换网上的不同主机提供通信服务。在发送数据时候，网络层把运输层产生的报文段或用户数据报封装成分组或IP数据包进行传送。并选择合适的路由，使传输层传下来的分组能够找到找到目标主机。

  • 数据链路层

    为同一链路中的主机提供数据点对点传输服务，在两个相邻结点之间传送数据时，数据链路层将网络层交下来的IP数据报组装成帧（framing），在两个相邻结点间的链路上“透明”地传送帧（frame）中的数据。

  • 物理层

    在传输介质上透明的传输比特流，屏蔽传输媒体和通信手段的差异。

  • 会话层

    建立及管理会话

  • 表示层

    数据压缩、加密及数据描述，使得应用程序不必关系数据格式不同的问题。

2. 三次握手过程

TCP三次握手是可靠传输的基础，主要是为了确认双方的接收和发送能力正常、同步双方的序列号和确认号并交换TCP窗口大小信息。

  • 第一次握手(SYN = 1, seq = x)

    客户端发送一个带有SYN = 1标记的数据包，并指明初始化序列号seq = x，客户端进入SYN_SENT阶段。

    需要消耗序号且不能携带数据。

  • 第二次握手(SYN = 1, ACK = 1, seq = y, ack = x + 1)

    服务端收到客户端的SYN报文后，可以知道客户端的发送正常、自己的接收正常，会回复一个带有SYN = 1、ACK = 1标记的数据包，并带上自己的序列号seq = y和确认号ack = x + 1，服务端进入SYN-RCVD阶段。

    需要消耗序号且不能携带数据。

  • 第三次握手(ACK = 1, seq = x + 1, ack = y + 1)

    客户端收到SYN报文后，可以知道自己和对方的发送、接收都正常，会回复一个带有ACK = 1的确认数据包，并带上自己的序列号seq = x + 1和回复确认号ack = y + 1，客户端进入ESTABLISHED阶段。

    服务端收到ACK确认报文后，会得知自己和对方的发送、接受正常，也会进入ESTABLISHED阶段，TCP握手结束。

    需要消耗序号，允许携带数据。

3. 为什么需要三次握手，两次不行吗？

  1. 三次握手实现了双方确认自己和对方的接收和发送正常。

    第一次握手，客户端发送，服务端接收，服务端可以知道：客户端的发送正常、服务端的接收正常；

    第二次握手，服务端发送，客户端接收，客户端可以知道：服务端的发送、接收正常，客户端的发送、接收正常，但此时服务端并不能确定自己的发送和客户端的接收是否正常。

    第三次握手，客户端发送，服务端接收，服务端可以知道：客户端的发送、接收正常，服务端的发送、接收正常。

    所以，三次握手才可以确认双方的发送、接收正常。

  2. 防止失效连接请求到达服务端，打开错误连接

    如果客户端发送的第一次握手请求在网络中滞留，客户端等待一段时间后会超时重传。这样客户端发送了两个连接请求，第一个滞留，第二个到达服务端成功建立连接。但是第一个滞留请求最后到达了服务器，如果没有第三次握手就会建立两个连接，而如果有第三次握手，客户端会忽略滞留请求发送的连接确认，不会再次打开连接。

4. 半连接队列

服务器收到第一次握手的SYN数据包后，处于SYN_RVCD状态，此时还没建立连接，服务器会把这种状态下的请求连接放在半连接队列中。

如果服务端发送第二次握手后未收到客户端确认包，就会进行超时重传，重传次数超过最大重传次数，就将此连接从半连接队列中删除。

5. SYN攻击

SYN攻击就是Client在短时间内伪造大量不存在的IP地址，并向Server不断地发送SYN包，Server则回复确认包，并等待Client确认，由于源地址不存在，因此Server需要不断重发直至超时，这些伪造的SYN包将长时间占用未连接队列，导致正常的SYN请求因为队列满而被丢弃，从而引起网络拥塞甚至系统瘫痪。SYN 攻击是一种典型的 DoS/DDoS 攻击。

在 Linux上可以使用系统自带的 netstat 命令来检测 SYN 攻击。

有以下几种防范手段：

  • 缩短超时时间

  • 增加半连接队列容量

  • 过滤网关防护

  • SYN cookies

    SYN cookies 技术可以使服务器在半连接队列已满的情况下仍能处理新的 SYN 请求。在半连接队列满时，不丢弃SYN请求，而是将SYN请求进行hash运算，并加密后得到服务端的初始序列号，称为cookie。服务器端在发送SYN + ACK包后，释放分配的连接请求块，如果收到客户端的ACK包，将客户端的ack序列号减一得到的值与hash运算得到的值进行比较，如果相等，直接完成三次握手。

    SYN cookies 机制的核心就是避免攻击造成的大量构造无用的连接请求块，导致内存耗尽，而无法处理正常的连接请求。

6. TCP KeepAlive

建立连接后，交互双方突然发生故障，导致没有正常释放TCP连接，会通过TCP KeepAlive机制释放无效资源。

TCP KeepAlive 的基本原理是，隔一段时间给连接对端发送一个探测包，如果收到对方回应的 ACK，则认为连接还是存活的，在超过一定重试次数之后还是没有收到对方的回应，则丢弃该 TCP 连接。

TCP KeepAlive发送的探测包会给网络带来额外的流量，并且只能在内核层级监测连接的存活与否，不能代表服务的可用性。需要保活的话需要在应用层实现自己的心跳机制。

7. 四次挥手过程

因为TCP的半关闭特性，TCP连接的关闭需要发送四个数据包，客户端与服务端均可主动发起挥手。

  • 第一次挥手(FIN = 1, seq = u)

    客户端发送连接释放报文，发送一个带有FIN = 1标记的数据包，并带上自己的序列号seq = u，发送后客户端进入FIN_WAIT1阶段。

  • 第二次挥手(ACK = 1, seq = v, ack = u + 1)

    服务端收到客户端的FIN包，回复一个ACK = 1确认包，并带上自己的序列号seq = v、确认号ack = u + 1，表明收到客户端的关闭连接请求。发送后服务端进入CLOSE_WAIT状态，客户端收到后进入FIN_WAIT2状态。

    此时TCP连接处于半关闭状态，服务端通知应用程序客户端到服务端的连接释放，但服务端仍能向客户端发送数据。

  • 第三次挥手(FIN = 1, ACK = 1, seq = w, ack = u + 1)

    等到服务端的数据传输完毕，向客户端发送带有FIN = 1的连接释放请求，并带上自己的序列号seq = w，确认号ack = u + 1，服务端进入LAST_ACK状态，等待来自客户端的确认信号。

  • 第四次挥手(ACK = 1, seq = u + 1, ack = w + 1)

    客户端收到FIN后，会回复一个ACK = 1确认报文，并带上序列号seq = u + 1，确认号ack = w + 1，并进入TIME_WAIT状态，等待2MSL后进入CLOSED状态；服务端接收到确认包后，关闭连接，进入CLOSED状态。

8. 为什么建立连接需要三次握手，关闭连接需要四次挥手？

建立连接时可以把SYN和ACK放在一个报文里发送，ACK用来应答，SYN用于同步。

但关闭连接时，服务端收到客户端的FIN报文，表示客户端已经没有数据要传送了，客户端到服务端的连接可以关闭，需要先回复一个ACK确认报文；但服务端可能还有数据要发送，需要等到服务端数据发送完成再发送FIN报文。FIN和ACK需要分开发送，所以需要四次挥手。

9. 为什么 TIME_WAIT 状态还需要等 2MSL 后才能进入CLOSED 状态？

MSL指的是任何报文段在网络中存活的最长时间，超过这个时间的报文会被丢弃。

TIME_WAIT状态等待2MSL，是为了：

  1. 保证客户端发送的最后一个ACK确认报文能够到达服务端

    客户端发送的ACK确认报文可能丢失，处于LAST_ACK状态的服务端没有收到确认报文，会进行超时重发，客户端能在2MSL内收到重发的FIN报文，重新启动2MSL计时器。

  2. 防止“已失效的连接请求报文段”出现在下一次新连接中

    客户端在发送完最后一个ACK报文段后，再经过2MSL，就可以使本连接持续的时间内所产生的所有报文段都从网络中消失，使下一个新的连接中不会出现这种旧的连接请求报文段。

10. TCP与UDP的区别

  1. TCP面向连接，UDP是无连接的，即发送数据前不需要建立连接
  2. TCP提供了确认、超时重传、滑动窗口、拥塞控制等机制来实现可靠传输，而UDP不提供可靠交付
  3. TCP面向字节流，UDP面向报文
  4. TCP连接只支持点对点，而UDP支持一对一、一对多和多对多的通信
  5. TCP的首部20字节，UDP首部大小8字节，TCP的开销大于UDP
  6. TCP适用于文件传输、发送接收邮件、远程登陆等场景，UDP适用于即时通信等场景。

11. TCP如何保证可靠传输

  • 三次握手，保证两端建立连接
  • 应用数据被分割成TCP认为最适合发送的数据块，将数据截断为合理长度
  • 校验和。TCP将保持首部和数据的校验和，目的是检测数据在传输过程中的任何变化。如果收到的段的检验和有差错，TCP将丢弃这个报文段和不确认收到此报文段。（不给出响应，超时重传）
  • 丢弃重复。TCP接收端会抛弃重复的数据
  • 乱序重排。TCP给发送的每一个包进行编号，接收方对数据包进行排序，把有序数据传送给应用层
  • 超时重传。TCP发送一个数据段后，启动一个定时器，等待目的端确认收到这个报文段。如果不能及时收到一个确认，将重发这个报文段。
  • 流量控制。TCP连接双方都维护一个滑动窗口来代表发送和接收的缓冲空间，接收端只允许发送端发送接收端缓冲区能容纳的数据，通过接收端滑动窗口的大小来控制发送端发送的速率，防止包丢失。
  • 拥塞控制。当网络拥塞时，减少数据包的发送。拥塞控制包括慢开始、拥塞避免、快重传和快恢复。

12. 流量控制

TCP利用滑动窗口（连续ARQ协议）实现流量控制。流量控制是为了控制发送方发送速率，保证接收方能及时接收。

接收方回复的ACK确认报文中的窗口字段可以用来控制发送方窗口大小，从而影响发送方的发送速率。

流量控制引发的死锁：发送者收到了一个滑动窗口为0的应答，发送者停止发送，等待接受者的下一个应答。如果窗口不为0的应答丢失，则双方互相等待，造成死锁。

为了避免死锁，TCP使用了连续计时器。每当发送方收到一个零窗口应答就启动计时器，时间一到就主动发送报文询问窗口的大小。如果仍是零窗口，则重置计时器继续等待；如果不为零，则更新发送窗口，继续发送。

13. 拥塞控制

拥塞控制是为了防止将过多的数据发送到网络中，造成网络的拥塞和过载。

流量控制是为了让接收方来得及接收，防止分组丢失；拥塞控制是为了降低网络的拥塞程度。

发送方维持一个拥塞窗口cwnd的状态变量，拥塞窗口的大小取决于网络的拥塞程度，并动态变化。发送方让自己的发送窗口取为拥塞窗口和接收方的接收窗口中较小值。TCP拥塞控制分为慢开始、拥塞避免、快重传和快恢复四个算法。

  • 慢开始

    慢开始指的是从小到大逐渐增加拥塞窗口的大小。开始时将cwnd设置为1，每经过一个传输轮次就将拥塞窗口cwnd加倍，直到到达慢开始门限ssthresh。当cwnd > ssthresh时，进入拥塞避免阶段。

    一个传输轮次所经历的时间就是往返时间RTT

  • 拥塞避免

    拥塞避免阶段让拥塞窗口缓慢增长，每经过一个往返时间RTT就把拥塞窗口cwnd加一，使拥塞窗口线性缓慢增长。

    在慢开始和拥塞避免阶段，如果出现超时（没有按时收到确认），就把慢开始门限ssthresh设为当前拥塞窗口的一半，即ssthresh = cwnd / 2，再重新执行慢开始。

  • 快重传

    快重传指的是接收方每次收到报文段时都会对最后一个已收到的有序报文段进行确认，如果发送方收到三个重复确认就会立即重发对方尚未收到的下一个报文段。

  • 快恢复

    快恢复指的是丢失个别报文段而不是网络阻塞时，并不重新执行慢开始，而是将拥塞窗口cwnd设置为当前ssthresh减半后的值，即cwnd = ssthresh / 2，然后执行拥塞避免使cwnd缓慢增大。

14. TCP与UDP报文首部

  • UDP首部格式

UDP报文首部包括源端口、目的端口、长度和校验和。

  • TCP首部格式

    • 序号：用于对字节流进行编号
    • 确认号：期望收到的下一个报文段的序号
    • 源端口和目的端口：指示终端中的不同进程
    • 数据偏移（首部长度）：数据部分距离报文段起始处的偏移量。
    • 标志位：
      • URG = 1，指示报文中有紧急数据，应尽快传送
      • SYN = 1，处于TCP连接建立过程
      • ACK = 1，确认序号（AN）有效
      • FIN = 1，请求释放连接
    • 窗口：滑动窗口，用于流量控制
    • 校验和：校验报文首部、数据
    • 紧急指针：指示紧急数据的大小

15. DNS过程

DNS是应用层协议，用于从域名解析出IP地址。

域名具有层次结构，从上到下分为：根域名、顶级域名、二级域名。

DNS流程：

  1. 在浏览器中输入域名，操作系统会先检查自己本地的hosts文件是否有这个网址映射关系，如果有，就先调用这个IP地址映射，完成域名解析
  2. 如果hosts里没有这个域名的映射，则查找本地DNS解析器缓存，是否有这个网址映射关系，如果有，直接返回，完成域名解析
  3. 如果本地DNS缓存中也没有，则去查询本地DNS服务器，如果要查询的域名，包含在本地配置区域资源中，则返回解析结果给客户机，完成域名解析，此解析具有权威性；如果不由本地DNS服务器区域解析，但该服务器已缓存了此网址映射关系，则调用这个IP地址映射，完成域名解析，此解析不具有权威性。
  4. 如果本地DNS服务器本地区域文件与缓存解析都失效，则按照是否设置转发进行查询：
    • 设置转发模式，将请求转发给上一级DNS服务器
    • 非转发模式，把请求发送给13台根DNS服务器，按照根DNS->顶级域名DNS->二级域名DNS服务器进行迭代查询。

从客户端到本地DNS服务器是属于递归查询，而DNS服务器之间就是的交互查询就是迭代查询。

17. ARQ协议

ARQ协议指的是自动重传请求，是数据链路层和传输层的错误纠正协议之一。通过使用确认和超时两个机制，在不可靠服务的基础上实现可靠信息传输。如果发送方一段时间内没有收到确认，会重新发送。

ARQ分为停止等待ARQ协议和连续ARQ协议。

停止等待ARQ

停止等待ARQ的基本原理是每发完一个分组就停止发送等待确认ACK。如果一段时间后还没有收到ACK确认，就重新发送，直到收到确认再发送下一个分组。

优点是简单，缺点是信道利用率低，等待时间长。

连续ARQ

发送方维持一个发送窗口，位于发送窗口内的分组可以连续发送出去，不需要等待接收方确认。

接收方采用累计确认，对按序到达的最后一个分组发送确认，表明到这个分组为止的所有分组都收到了。

优点是信道利用率高，缺点是无法获取接收方正确收到的分组信息。

连续ARQ错误纠正采用回退N重传的方式：

  • 接受方只对按序到达的最后一个分组发送确认
  • 发送方会对最后一个收到确认的分组的下一个分组开始重新发送

18. ARP协议

ARP协议是局域网下的地址解析协议，通过解析IP地址得到MAC地址，主要应用在ipv4中。

同一局域网中的一台主机要和另一台主机进行直接通信，必须要知道目标主机的MAC地址。而网络层和传输层只关心目标主机的IP地址，所以需要通过ARP协议将目标IP地址转化为目标MAC地址。

当两台主机不在同一局域网内时，即使知道MAC地址也不能直接通信，必须经过路由转发。此时发送主机通过ARP获取的就是通往局域网外路由器的MAC地址。

每个主机都有ARP高速缓存，同一局域网下发送数据时，会先查看缓存中是否有目标IP到MAC地址的映射。如果没有，会通过广播的方式发送ARP请求分组，目标主机收到后会回复ARP响应，发送主机收到后会在缓存中记录此映射关系。

19. ICMP协议

网络控制消息协议，主要用于传递查询报文和差错报文，属于网络层协议。

ICMP的功能主要有：确认IP包是否到达目标地址和通知在发送过程中IP包被丢弃原因。

ICMP主要有两个应用：Ping和TraceRoute。

Ping

ping是ICMP的一个重要应用，主要用于测试两台主机间的连通性。

源主机通过向目标主机间发送ICMP请求报文，目标主机收到后回复应答报文。Ping通过时间和成功响应的次数估算往返时间和丢包率。

TraceRoute

TraceRoute用于跟踪一个分组从源点到终点的路径，原理是利用ICMP差错控制报文中的TTL超时会回向源点发送一个时间超时报文。

初始时将发送分组的TTL置为1，并每次递增1。分组每到达一个路由器，TTL都会减一，当TTL为0就会向源主机发送时间超时报文。当到达目标主机时，就获得了从源点到终点的路径。

20. HTTP 首部 && 状态码

HTTP首部

HTTP报文由请求行/状态行、通用首部字段、请求首部字段/响应首部字段和实体首部字段等构成

请求行包括HTTP方法、请求uri和HTTP版本。

状态行包括HTTP版本和响应状态码。

HTTP响应码

HTTP响应首部的状态行中给出了响应码表明了请求的结果。

  • 1XX 信息

    • 100 Continue：表明到现在为止都很正常，客户端应该继续请求，如果已经完成，则忽略它。
    • 102 Processing：表示服务器已收到并正在处理该请求，但没有响应可用。

  • 2XX 成功

    • 200 OK：请求成功。
    • 201 Created：请求成功，创建了一个新的资源。常用于POST请求。
    • 202 Accepted：请求已收到但还未响应，没有结果。意味着不会有一个异步的响应去表明当前请求的结果。
    • 204 No Content：请求成功处理，但返回的响应报文不包含实体内容。
    • 206 Partial Content：表示客户端进行了范围请求，响应报文包含指定范围的实体内容。

  • 3XX 重定向

  • 301 Moved Permanently：永久重定向

    • 302 Found：临时重定向
    • 303 See Other：与302相同，但303表示客户端应该采用GET获取资源
    • 304 Not Modified：如果请求首部包含一些条件，条件不满足时返回304状态码
    • 307 Temporary Redirect：临时重定向，但不会要求浏览器把重定向请求的POST改为GET

  • 4XX 客户端错误

    • 400 Bad Request：请求报文中存在语法错误
    • 401 Unauthorized：当前请求需要用户认证
    • 403 Forbidden：请求被拒绝
    • 404 Not Found：请求资源未被发现

  • 5XX 服务端错误

    • 500 Internal Server Error：服务器执行请求时发生错误
    • 502 Bad Gateway：网关得到了错误响应
    • 503 Service Unavailable：服务器过载或在停机维护，无法处理请求

21. HTTP 与 HTTPS的区别

  1. 端口

    HTTP默认运行在80端口，HTTPS默认使用443端口。

  2. 安全性和资源消耗

    HTTP协议运行在TCP上，使用明文传输。而HTTPS是运行在SSL/TLS上的HTTP协议，传输内容经过对称密钥加密，而对称加密的密钥使用非对称密钥加密传输。因为非对称加密计算量较大，整个通信过程中只会在传输对称加密密钥时采用非对称加密，内容的加密都是采用对称加密。

HTTPS通信前会向进行一次SSL握手，流程如下：

  1. 客户端向服务端的443端口发起请求，并携带客户端支持的加密算法和哈希算法；
  2. 服务端收到请求，选择客户端支持的加密算法和哈希算法，并将数字证书返回给客户端；
  3. 浏览器进行证书认证，利用机构的证书公钥解密得到证书内容和证书签名，内容包括网站的公钥、证书有效期等。接着浏览器会进行证书签名的认证，浏览器利用网站公钥解密证书签名得到一串哈希值，然后对证书内容进行同样的不可逆哈希算法处理也会得到一串哈希值，对比两次哈希值，如果相等则证书签名认证通过。
  4. 浏览器产生一个随机数R，并使用网站公钥对R进行加密，然后发送给服务器
  5. 服务器用自己的私钥解密得到R
  6. 之后进行通信时以R为密钥对内容进行对称加密。

22. GET 与 POST的区别

  1. 作用：GET和POST都是HTTP方法，GET用于获取资源，POST用于提交资源。
  2. 参数：GET和POST请求都可以携带参数，GET的参数一般出现在URL中，POST的参数存储在实体主体中。因为URL有长度限制且支持ASCII码，所以GET方法参数也有长度限制且中文会被替换为ASCII码；POST方法支持标准字符集。
  3. 安全：GET和POST都不是安全的，都可以通过抓包来获取参数
  4. 幂等性：GET方法是幂等的，而POST不是幂等的。
  5. 可缓存：GET方法可缓存，而POST不可以。

23. HTTP 1.0、1.1、2.0

  1. 长连接。HTTP 1.0中默认使用的是短连接，每次请求都要重新进行三次握手和四次挥手，开销较大；从HTTP 1.1开始，默认使用长连接，默认开启Connection： keep-alive
  2. 流水线。HTTP1.0 中每次要收到上一次请求的响应才能发送下一个请求，HTTP 1.1中引入了流水线方式，在收到响应报文前可以接着发送新的请求报文。
  3. 错误响应码。HTTP1.1中新增了24个错误响应码。
  4. 缓存处理。在HTTP1.0中主要使用header里的If-Modified-Since,Expires来做为缓存判断的标准，HTTP1.1则引入了更多的缓存控制策略例如Entity tag，If-Unmodified-Since, If-Match, If-None-Match等更多可供选择的缓存头来控制缓存策略。
  5. 带宽优化。HTTP1.0中，会把整个对象发送过来，在HTTP1.1中，在请求头引入了range头域，允许只请求资源的某个部分，此时返回码为206。

HTTP2.0 新增了二进制分帧、服务端推送、header压缩和多路复用机制。

  • 二进制分帧。HTTP1.x的解析基于文本，而HTTP2.0中 headers和data中都是二进制数据

  • 服务端推送。客户端请求资源时，服务端会把相关资源一起发送给客户端。

  • header压缩。HTTP1.x的header中带有大量信息，每次都要重新发送。HTTP2.0中客户端和服务端各自维护一份首部字段表，避免重复传输，同时使用压缩算法压缩首部字段。

  • 多路复用。连接共享，一个连接上可以有多个request，接收方根据request的id将request归属到各自不同的服务端请求中。
