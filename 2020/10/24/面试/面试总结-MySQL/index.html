面试总结-MySQL

发表于 2020-10-24 | 分类于 面试

1. 数据库引擎MyISAM和InnoDB区别

MyISAM在5.5以前是MySQL的默认数据库引擎，支持全文索引、压缩和空间函数等，但MyISAM不支持事务和行级锁，并且崩溃后无法安全恢复。5.5之后，MySQL默认使用InnoDB作为默认数据库引擎。

对比：

  • 是否支持行级锁：MyISAM只支持表级锁，而InnoDB支持表级锁和行级锁，默认为行级锁；
  • 是否支持事务和崩溃后安全恢复：MyISAM追求查询效率，不支持事务；而InnoDB提供事务支持，具有事务回滚和崩溃后修复能力。
  • 是否支持外键：MyISAM不支持，InnoDB支持
  • 是否支持MVCC：只有InnoDB支持
  • MyISAM支持全文索引，InnoDB不支持
  • MyISAM相对简单，适合读密集场景；而InnoDB适合大型读写高并发场景。

2. MySQL中一条语句的执行过程&&MySQL架构

MySQL主要分为Server层和存储引擎层。

  • Server层：主要包括连接器、查询缓存、分析器、优化器和执行器，还有一个通用的日志模块binlog；
  • 存储引擎：主要负责数据的读取和存储，采用可替换的插件式架构，支持InnoDB、MyISAM、Memory等存储引擎，5.5版本后默认使用InnoDB，其自带了一个日志模块redolog。

MySQL Server层基本组件

  • 连接器

    连接器主要负责身份认证和权限相关

  • 查询缓存（MySQL8.0后移除）

    查询缓存主要用来缓存执行的SELECT语句及其结果集。

    连接建立后，执行查询语句时会先查询缓存，以key-value形式缓存在内存中，key是查询语句，value是结果集。如果缓存命中，直接返回结果，如果没有命中，再执行后续操作，完成后会将结果缓存起来。

    查询缓存在MySQL8.0后被移除

  • 分析器

    分析器主要是用来分析SQL语句是用来干嘛的，分为两步：

    1. 词法分析：提取关键字，如select、表名、查询条件等；
    2. 语法分析：判断SQL语句是否有错误

  • 优化器

    优化器的作用就是按照MySQL认为最优的方案去执行（可能不是最优）

  • 执行器

    执行器在执行前会判断用户是否有权限，如果没有返回错误信息，如果有权限，就会调用存储引擎的接口，并返回执行结果。

执行过程

  • 查询语句

    连接器权限校验->查询缓存->分析器->优化器->权限校验->执行器->存储引擎查询

  • 更新语句

    连接器权限校验->分析器->权限校验->执行器->存储引擎–redo log prepare->bin log -> redo log commit

    先redo log预提交，再写bin log，最后redo log提交是为了保证数据一致性。

3. 事务特性及其实现

事务是逻辑上的一组操作，要么全部执行，要么全部回滚。

MySQL事务有四大特性(ACID)：原子性、一致性、隔离性、持久性

原子性

事务是一系列原子操作，要么全部完成，要么全部回滚

MySQL实现事务原子性的基础是undo log。

事务回滚指的是能够撤销所有已经成功执行的SQL语句。当事务对数据库进行修改时，会把所有的写操作写入undo log，当进行回滚时，利用undo log中的日志信息做相反的逻辑，将数据回滚到修改之前的版本。

原子性只能保证单个事务的一致性，并不能保证多线程下的事务一致性。

一致性

事务前后数据库总是从一个状态到达另一个状态，一致性是最基本的属性，其他三个性质都是为了保证一致性。

原子性只能保证单线程下的一致性，所以引入了隔离性来保证并发下的一致性。

隔离性

并发访问时，各个事务间是相互独立的。MySQL中定义了四种隔离级别：

数据库中主要是通过多版本并发控制(MVCC)和锁来实现隔离性的。

持久性

事务被提交后，对数据库的改变是持久的，即使发生了故障也不会产生影响。

MySQL通过redo log来实现持久性。当做数据修改时，不仅在内存中操作，还会在redo log中记录这次操作。当事务提交的时候，会将redo log进行刷盘。当宕机重启时，会将redo log中的内容恢复到数据库中，再根据undo log和bin log内容决定回滚数据还是提交数据。

4. 并发事务的问题

并发事务可能会造成：脏读、丢失修改、不可重复读和幻读。

  • 脏读：一个事务可以读取到其他事务未提交的脏数据
  • 丢失修改：两个事务并发修改同一个数据，导致第一个事务的修改结果丢失
  • 不可重复读：一个事务多次读取同一数据，期间另一事务修改了该数据，造成事务两次读取的数据不一样。一个事务内两次读到的数据不一致就是不可重复读。
  • 幻读：第一个事务读取了几行数据，接着另一个事务插入了几行数据，随后第一个事务读取发现多出了一些原本不存在的记录，像是产生了幻觉，称为幻读。

不可重复读与幻读的区别：

不可重复读的重点是多次读取一条记录发现值被修改，而幻读的重点是数据列的新增或者删除。

5. 事务隔离级别

MySQL中有四个隔离级别，不同的隔离级别是在数据可靠性和并发性之间进行取舍，隔离级别越高，并发性能越差，数据越安全可靠。

READ-UNCOMMITTED（读取未提交）

最低的隔离级别，允许读取尚未提交的数据，可能会导致脏读、不可重复读、幻读。

READ-UNCOMMITED隔离级别下，读不会加任何锁，写会加排它锁，并在事务结束后释放。

排它锁会阻塞其他事务对其锁定的数据加读锁或写锁的操作，但读取未提交隔离级别下都不会加锁，所以对不加锁的读不起作用。

READ-COMMITED（读取已提交）：

只能读取并发事务已提交的数据，即一个事务所做的修改在提交之前是不可见的。可通过MVCC实现。可能会导致不可重复读和幻读。

InnoDB在READ-COMMITTED隔离级别下，写数据时加排它锁，读数据时不加锁而是采取MVCC多版本并发机制。

READ-COMMITTED隔离级别下，每次进行select都会生成MVCC版本，所以在select之间有其他事务更新了数据，就会产生不可重复读。

REPEATABLE-READ（可重复读）：

保证同一个事务中多次读取相同数据结果相同。通过MVCC实现。可能产生幻读。MySQL默认的隔离级别就是可重复读。

InnoDB在REPEATABLE-READ隔离级别下，写数据时加排它锁，读数据时也是采用的MVCC多版本并发机制，不过可重复读仅仅是在事务开始时创建一次快照。

SERIALIZABLE（可串行化）：

强制事务串行支持。需要加锁实现。

SERIALIZABLE可串行化隔离级别下，读数据时加共享锁，写数据时加排它锁。

6. 多版本并发控制MVCC

多版本并发控制MVCC是InnoDB实现隔离级别的一种具体方式，主要用于实现提交读和可重复读隔离级别。

事务的隔离性可以通过加锁来实现，但频繁地加锁会极大的降低数据库的并发性能。

MVCC本质上就是保存数据的快照，从而实现读取数据时不需要加锁就可以实现隔离性，是一种乐观锁的思想。

在InnoDB中，会在每行记录后保存两个隐藏的列，用来存储两个版本号：

  • 创建版本号：指示创建一个数据行的快照时的系统版本号
  • 删除版本号：如果该快照的删除版本号大于当前事务的系统版本号表示有效，否则表示该数据已经被删除了

其中，系统版本号是一个递增的数字，每开始一个新的事务，系统的版本号就自动递增；事务版本号是指事务开始时的系统版本号。

MVCC使用的快照存储在undo log中，其通过回滚指针把一个数据行的所有快照连接起来。

undo log主要有两个作用：

  1. 保证事务的原子性，事务回滚时利用uodo log进行恢复
  2. 用于存储MVCC快照读的数据，通过读取undo log的历史版本数据可以实现不同事务版本号都有自己独立的快照数据版本

在可重复读REREATABLE-READ隔离级别下，开始一个事务时，该事务的版本号肯定大于当前所有数据行快照的创建版本号。

  • select

    读取创建版本号<=当前事务版本号，删除版本号为空或者大于当前事务版本号。

  • insert

    将当前事务的版本号作为数据行快照的创建版本号

  • delete

    将当前事务的版本号作为数据行快照的删除版本号

  • update

    将当前事务的版本号作为更新前数据行的删除版本号，并将当前系统版本号作为更新后数据行快照的创建版本号。可以理解为先执行delete再执行insert

通过MVCC读取到的数据是历史数据，属于快照读，可以减少加锁的开销；而读取数据库当前版本的数据叫做当前读，都需要加锁。

  • 快照读：读取的是历史数据，不需要加锁

  • 当前读：读取的是当前的数据，特殊的读操作、增、删、改操作，属于当前读，需要加锁

提交读和可重复读都可以使用MVCC来实现，不同的是提交读隔离级别下每次select都会重新创建快照，而可重复读隔离级别下只会在事务开始时创建一次快照。

使用MVCC可以解决不可重复读问题，但无法解决幻读。为了解决当前读中的幻读问题，MySQL使用了Next-Key Lock锁。

7. 锁分类与InnoDB锁算法

锁分类

  1. 按照锁的粒度分

    按照锁的粒度，MySQL锁可以分为表锁和行锁。

    • 表级锁

      对当前操作的整张表加锁，锁定力度最大，实现简单，加锁快，不会出现死锁，但触发锁冲突的概率最大，并发度最低。MyISAM支持只表级锁，InnoDB也支持表级锁。

    • 行级锁

      只对当前操作的行加锁，锁定粒度最小，行级锁能大大减少数据库的冲突，并发度高，但加锁开销大，加锁慢，会出现死锁。InnoDB支持行级锁。

      InnoDB支持的行级锁包括：Record Lock、Gap Lock和Next-key Lock。

    • 页级锁

      页级锁的锁定力度介于表级锁和行级锁之间，一次锁定相邻的一组记录。BDB引擎支持页级锁。

  1. 按照锁的思想分类

    按照锁的思想，可以分为乐观锁和悲观锁

    乐观锁会假定大概率不会发生冲突，访问、处理数据过程中不加锁，只在更新数据时根据数据版本号或时间戳判断是否有冲突，有则处理，没有则提交事务。

    悲观锁会假定大概率会发生并发更新冲突，访问、处理数据前就会加排它锁，事务提交或回滚后才会释放锁。

  1. 按照锁的角色分类

    按照锁的角色，可以分为共享锁与排它锁

    • 共享锁

      共享锁，又被称为读锁，一个事务对数据加了共享锁，可以对数据进行读取但不能更新，其他事务能对这个数据加共享锁，但不能加排它锁。

    • 排它锁

      排它锁，又被称为写锁，一个事务对数据加了排它锁，可以对数据进行读取和更新，其他事务不能加任何锁。

  1. 意向锁

    使用意向锁可以更好的实现多粒度封锁，意向锁是一个不与行级锁冲突的表级锁。

    事务想对表加锁时，需要先检测是否有其他事务对表加过了与自己冲突的锁，需要对表的每一行都检测一次，非常耗时。所以MySQL在原来X/S的基础上引入了意向共享锁(IS)和意向排它锁(IX)，在加锁前只需检查表上的意向锁。

    • 意向共享锁(IS)

      事务要获取某些行的S锁，必须先获得表的IS锁。

      SELECT column FROM table ... LOCK IN SHARE MODE;

    • 意向排它锁(IX)

      事务要获取某些行的X锁，必须先获得表的IX锁。

      SELECT column FROM table ... FOR UPDATE;

    各种锁之间的兼容关系如下：

    - X IX S IS
    X × × × ×
    IX × √ × √
    S × × √ √
    IS × √ √ √

    解释：

    • 任意IS/IX之间都是兼容的，因为他们都是意向锁，而不是真正加锁
    • S锁只与S锁和IS锁兼容

锁算法

InnoDB所有的行锁算法都是基于索引实现的，锁定的都是索引或者索引区间。使用主键索引时，会锁定主键索引；使用非主键索引时，会先锁住非主键索引，再锁住主键索引。

  • Record Lock

    记录锁，锁定一个行记录的索引。

  • Gap Lock

    间隙锁，锁定一个索引范围，不包括记录本身

  • Next-key Lock

    相当于Record Lock加上Gap Lock，锁定一个范围，包括记录本身。

    使用Next-key Lock，InnoDB可以在可重复读隔离级别下解决幻读问题。

总结：

  1. InnoDB对于行的查询使用Next-key Lock；
  2. Next-key Lock可以解决幻读问题
  3. 当查询的索引具有唯一性时，会将next-key Lock降级为Gap Lock
  4. Gap锁设计的目的是防止多个事务将记录插入到同一范围，导致幻读
