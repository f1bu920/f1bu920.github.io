面试总结-MySQL

发表于 2020-10-24 | 分类于 面试

1. 数据库引擎MyISAM和InnoDB区别

MyISAM在5.5以前是MySQL的默认数据库引擎，支持全文索引、压缩和空间函数等，但MyISAM不支持事务和行级锁，并且崩溃后无法安全恢复。5.5之后，MySQL默认使用InnoDB作为默认数据库引擎。

对比：

  • 是否支持行级锁：MyISAM只支持表级锁，而InnoDB支持表级锁和行级锁，默认为行级锁；
  • 是否支持事务和崩溃后安全恢复：MyISAM追求查询效率，不支持事务；而InnoDB提供事务支持，具有事务回滚和崩溃后修复能力。
  • 是否支持外键：MyISAM不支持，InnoDB支持
  • 是否支持MVCC：只有InnoDB支持
  • MyISAM支持全文索引，InnoDB不支持
  • MyISAM相对简单，适合读密集场景；而InnoDB适合大型读写高并发场景。

2. MySQL中一条语句的执行过程&&MySQL架构

MySQL主要分为Server层和存储引擎层。

  • Server层：主要包括连接器、查询缓存、分析器、优化器和执行器，还有一个通用的日志模块binlog；
  • 存储引擎：主要负责数据的读取和存储，采用可替换的插件式架构，支持InnoDB、MyISAM、Memory等存储引擎，5.5版本后默认使用InnoDB，其自带了一个日志模块redolog。

MySQL Server层基本组件

  • 连接器

    连接器主要负责身份认证和权限相关

  • 查询缓存（MySQL8.0后移除）

    查询缓存主要用来缓存执行的SELECT语句及其结果集。

    连接建立后，执行查询语句时会先查询缓存，以key-value形式缓存在内存中，key是查询语句，value是结果集。如果缓存命中，直接返回结果，如果没有命中，再执行后续操作，完成后会将结果缓存起来。

    查询缓存在MySQL8.0后被移除

  • 分析器

    分析器主要是用来分析SQL语句是用来干嘛的，分为两步：

    1. 词法分析：提取关键字，如select、表名、查询条件等；
    2. 语法分析：判断SQL语句是否有错误

  • 优化器

    优化器的作用就是按照MySQL认为最优的方案去执行（可能不是最优）

  • 执行器

    执行器在执行前会判断用户是否有权限，如果没有返回错误信息，如果有权限，就会调用存储引擎的接口，并返回执行结果。

执行过程

  • 查询语句

    连接器权限校验->查询缓存->分析器->优化器->权限校验->执行器->存储引擎查询

  • 更新语句

    连接器权限校验->分析器->权限校验->执行器->存储引擎–redo log prepare->bin log -> redo log commit

    先redo log预提交，再写bin log，最后redo log提交是为了保证数据一致性。

3. 事务特性及其实现

事务是逻辑上的一组操作，要么全部执行，要么全部回滚。

MySQL事务有四大特性(ACID)：原子性、一致性、隔离性、持久性

原子性

事务是一系列原子操作，要么全部完成，要么全部回滚

MySQL实现事务原子性的基础是undo log。

事务回滚指的是能够撤销所有已经成功执行的SQL语句。当事务对数据库进行修改时，会把所有的写操作写入undo log，当进行回滚时，利用undo log中的日志信息做相反的逻辑，将数据回滚到修改之前的版本。

原子性只能保证单个事务的一致性，并不能保证多线程下的事务一致性。

一致性

事务前后数据库总是从一个状态到达另一个状态，一致性是最基本的属性，其他三个性质都是为了保证一致性。

原子性只能保证单线程下的一致性，所以引入了隔离性来保证并发下的一致性。

隔离性

并发访问时，各个事务间是相互独立的。MySQL中定义了四种隔离级别：

数据库中主要是通过多版本并发控制(MVCC)和锁来实现隔离性的。

持久性

事务被提交后，对数据库的改变是持久的，即使发生了故障也不会产生影响。

MySQL通过redo log来实现持久性。当做数据修改时，不仅在内存中操作，还会在redo log中记录这次操作。当事务提交的时候，会将redo log进行刷盘。当宕机重启时，会将redo log中的内容恢复到数据库中，再根据undo log和bin log内容决定回滚数据还是提交数据。

4. 并发事务的问题

并发事务可能会造成：脏读、丢失修改、不可重复读和幻读。

  • 脏读：一个事务可以读取到其他事务未提交的脏数据
  • 丢失修改：两个事务并发修改同一个数据，导致第一个事务的修改结果丢失
  • 不可重复读：一个事务多次读取同一数据，期间另一事务修改了该数据，造成事务两次读取的数据不一样。一个事务内两次读到的数据不一致就是不可重复读。
  • 幻读：第一个事务读取了几行数据，接着另一个事务插入了几行数据，随后第一个事务读取发现多出了一些原本不存在的记录，像是产生了幻觉，称为幻读。

不可重复读与幻读的区别：

不可重复读的重点是多次读取一条记录发现值被修改，而幻读的重点是数据列的新增或者删除。

5. 事务隔离级别

MySQL中有四个隔离级别，不同的隔离级别是在数据可靠性和并发性之间进行取舍，隔离级别越高，并发性能越差，数据越安全可靠。

READ-UNCOMMITTED（读取未提交）

最低的隔离级别，允许读取尚未提交的数据，可能会导致脏读、不可重复读、幻读。

READ-UNCOMMITED隔离级别下，读不会加任何锁，写会加排它锁，并在事务结束后释放。

排它锁会阻塞其他事务对其锁定的数据加读锁或写锁的操作，但读取未提交隔离级别下都不会加锁，所以对不加锁的读不起作用。

READ-COMMITED（读取已提交）：

只能读取并发事务已提交的数据，即一个事务所做的修改在提交之前是不可见的。可通过MVCC实现。可能会导致不可重复读和幻读。

InnoDB在READ-COMMITTED隔离级别下，写数据时加排它锁，读数据时不加锁而是采取MVCC多版本并发机制。

READ-COMMITTED隔离级别下，每次进行select都会生成MVCC版本，所以在select之间有其他事务更新了数据，就会产生不可重复读。

REPEATABLE-READ（可重复读）：

保证同一个事务中多次读取相同数据结果相同。通过MVCC实现。可能产生幻读。MySQL默认的隔离级别就是可重复读。

InnoDB在REPEATABLE-READ隔离级别下，写数据时加排它锁，读数据时也是采用的MVCC多版本并发机制，不过可重复读仅仅是在事务开始时创建一次快照。

SERIALIZABLE（可串行化）：

强制事务串行支持。需要加锁实现。

SERIALIZABLE可串行化隔离级别下，读数据时加共享锁，写数据时加排它锁。

6. 多版本并发控制MVCC

多版本并发控制MVCC是InnoDB实现隔离级别的一种具体方式，主要用于实现提交读和可重复读隔离级别。

事务的隔离性可以通过加锁来实现，但频繁地加锁会极大的降低数据库的并发性能。

MVCC本质上就是保存数据的快照，从而实现读取数据时不需要加锁就可以实现隔离性，是一种乐观锁的思想。

在InnoDB中，会在每行记录后保存两个隐藏的列，用来存储两个版本号：

  • 创建版本号：指示创建一个数据行的快照时的系统版本号
  • 删除版本号：如果该快照的删除版本号大于当前事务的系统版本号表示有效，否则表示该数据已经被删除了

其中，系统版本号是一个递增的数字，每开始一个新的事务，系统的版本号就自动递增；事务版本号是指事务开始时的系统版本号。

MVCC使用的快照存储在undo log中，其通过回滚指针把一个数据行的所有快照连接起来。

undo log主要有两个作用：

  1. 保证事务的原子性，事务回滚时利用uodo log进行恢复
  2. 用于存储MVCC快照读的数据，通过读取undo log的历史版本数据可以实现不同事务版本号都有自己独立的快照数据版本

在可重复读REREATABLE-READ隔离级别下，开始一个事务时，该事务的版本号肯定大于当前所有数据行快照的创建版本号。

  • select

    读取创建版本号<=当前事务版本号，删除版本号为空或者大于当前事务版本号。

  • insert

    将当前事务的版本号作为数据行快照的创建版本号

  • delete

    将当前事务的版本号作为数据行快照的删除版本号

  • update

    将当前事务的版本号作为更新前数据行的删除版本号，并将当前系统版本号作为更新后数据行快照的创建版本号。可以理解为先执行delete再执行insert

通过MVCC读取到的数据是历史数据，属于快照读，可以减少加锁的开销；而读取数据库当前版本的数据叫做当前读，都需要加锁。

  • 快照读：读取的是历史数据，不需要加锁

  • 当前读：读取的是当前的数据，特殊的读操作、增、删、改操作，属于当前读，需要加锁

提交读和可重复读都可以使用MVCC来实现，不同的是提交读隔离级别下每次select都会重新创建快照，而可重复读隔离级别下只会在事务开始时创建一次快照。

使用MVCC可以解决不可重复读问题，但无法解决幻读。为了解决当前读中的幻读问题，MySQL使用了Next-Key Lock锁。

7. 锁分类与InnoDB锁算法

锁分类

  1. 按照锁的粒度分

    按照锁的粒度，MySQL锁可以分为表锁和行锁。

    • 表级锁

      对当前操作的整张表加锁，锁定力度最大，实现简单，加锁快，不会出现死锁，但触发锁冲突的概率最大，并发度最低。MyISAM支持只表级锁，InnoDB也支持表级锁。

    • 行级锁

      只对当前操作的行加锁，锁定粒度最小，行级锁能大大减少数据库的冲突，并发度高，但加锁开销大，加锁慢，会出现死锁。InnoDB支持行级锁。

      InnoDB支持的行级锁包括：Record Lock、Gap Lock和Next-key Lock。

    • 页级锁

      页级锁的锁定力度介于表级锁和行级锁之间，一次锁定相邻的一组记录。BDB引擎支持页级锁。

  1. 按照锁的思想分类

    按照锁的思想，可以分为乐观锁和悲观锁

    乐观锁会假定大概率不会发生冲突，访问、处理数据过程中不加锁，只在更新数据时根据数据版本号或时间戳判断是否有冲突，有则处理，没有则提交事务。

    悲观锁会假定大概率会发生并发更新冲突，访问、处理数据前就会加排它锁，事务提交或回滚后才会释放锁。

  1. 按照锁的角色分类

    按照锁的角色，可以分为共享锁与排它锁

    • 共享锁

      共享锁，又被称为读锁，一个事务对数据加了共享锁，可以对数据进行读取但不能更新，其他事务能对这个数据加共享锁，但不能加排它锁。

    • 排它锁

      排它锁，又被称为写锁，一个事务对数据加了排它锁，可以对数据进行读取和更新，其他事务不能加任何锁。

  1. 意向锁

    使用意向锁可以更好的实现多粒度封锁，意向锁是一个不与行级锁冲突的表级锁。

    事务想对表加锁时，需要先检测是否有其他事务对表加过了与自己冲突的锁，需要对表的每一行都检测一次，非常耗时。所以MySQL在原来X/S的基础上引入了意向共享锁(IS)和意向排它锁(IX)，在加锁前只需检查表上的意向锁。

    • 意向共享锁(IS)

      事务要获取某些行的S锁，必须先获得表的IS锁。

      SELECT column FROM table ... LOCK IN SHARE MODE;

    • 意向排它锁(IX)

      事务要获取某些行的X锁，必须先获得表的IX锁。

      SELECT column FROM table ... FOR UPDATE;

    各种锁之间的兼容关系如下：

    - X IX S IS
    X × × × ×
    IX × √ × √
    S × × √ √
    IS × √ √ √

    解释：

    • 任意IS/IX之间都是兼容的，因为他们都是意向锁，而不是真正加锁
    • S锁只与S锁和IS锁兼容

锁算法

InnoDB所有的行锁算法都是基于索引实现的，锁定的都是索引或者索引区间。使用主键索引时，会锁定主键索引；使用非主键索引时，会先锁住非主键索引，再锁住主键索引。

  • Record Lock

    记录锁，锁定一个行记录的索引。

  • Gap Lock

    间隙锁，锁定一个索引范围，不包括记录本身

  • Next-key Lock

    相当于Record Lock加上Gap Lock，锁定一个范围，包括记录本身。

    使用Next-key Lock，InnoDB可以在可重复读隔离级别下解决幻读问题。

总结：

  1. InnoDB对于行的查询使用Next-key Lock；
  2. Next-key Lock可以解决幻读问题
  3. 当查询的索引具有唯一性时，会将next-key Lock降级为Gap Lock
  4. Gap锁设计的目的是防止多个事务将记录插入到同一范围，导致幻读

8. 索引

8.1. 为什么要用索引

  • 通过创建唯一性索引，保证数据表中每一行数据的唯一性
  • 大大加快数据的检索速度
  • 避免排序和临时表
  • 将随机IO变为顺序IO
  • 加速表与表之间的连接

8.2. 索引如何提高检索速度

通过将无序的数据变为相对有序的数据提高检索速度，索引起到了类似目录的作用。

MySQL的基本存储结构是页：

各个数据页之间组成一个双向链表，每个数据页内部的记录组成一个单链表。当不使用索引时，只能先遍历双向链表找到对应的页，再遍历所在页的单链表，在数据量很大时查找会很慢。

而索引可以将无序的数据变成相对有序，查找时可以根据索引快速定位到所在的页。

8.3 索引越多越好？

  • 索引需要空间来存储，索引越多，占用的物理空间就越多
  • 索引越多，新增、更新和删除时的维护成本越大

8.4 索引的数据结构

MySQL中索引主要有两种数据结构，哈希索引和B+Tree索引。下面列出各种数据结构的对比：

  • 二叉搜索树

    左子树节点值小于根节点，右子树节点值大于根节点。在极端情况下会退化为链表，近乎全表扫描。

  • 红黑树

    红黑树本质上属于二叉平衡树，二叉平衡树的查找效率虽然很高，但维护代价很大，需要进行多次旋转来保证插入或更新后的平衡性。

  • 哈希索引

    哈希索引可以通过散列快速获取磁盘文件指针，等值查询非常快，但不支持范围查询

  • B-Tree

    目前大部分数据库都是采用B-Tree或者B+Tree作为索引结构。相比于其他各种树结构，B树和B+树能大幅降低树的高度，即能减少IO的次数。

    B树本质上属于多叉平衡树，叶子节点具有相同的深度，节点中数据索引从左到右递增：

B树有两个明显的特点：

树内索引节点也存储数据；叶子节点间没有指针连接。

  • B+Tree

    B+树是B树的变种，相比于B-Tree，主要有两个特点：

    数据只存储在叶子节点；所有叶子节点间通过指针连接。

8.5 B-Tree与B+Tree的对比

  1. B+Tree非叶子节点不存储数据，只存储索引，一页可以放更多的索引，降低了树高，根据局部性原理，按页加载到内存中，减少了IO次数
  2. B-Tree在单条数据查询时，查询效率不固定，最好情况是O(1)，可以认为单一数据查询B-Tree的平均性能更好
  3. B+Tree叶子节点间使用指针连接，提高了区间访问性能，非常适合范围查找。

因此，B+Tree非常适合做数据遍历操作，因此MySQL采用B+Tree作为默认索引结构；B-Tree适合单一查询较多范围查询较少的场景，因此MongDB采用B-Tree作为索引结构。

8.6 聚簇索引与非聚簇索引

  • 非聚簇索引

    非聚簇索引指的是B+Tree叶子节点data域存储的是数据记录的地址，按照索引找到对应的叶子节点后，再以data域的值为地址读取对应的数据记录。

    MyISAM引擎使用的就是非聚簇索引。

  • 聚簇索引

    聚簇索引的叶子节点保存的是完整的数据记录，并且数据文件本身就是主键索引按照B+Tree结构组织的。

    InnoDB引擎使用的就是聚簇索引。

    • 如果表中设置了主键，则主键就是聚簇索引；
    • 如果没有主键，就会默认第一个非空且唯一的列作为聚簇索引；
    • 以上都没有就会默认创建一个隐藏的row_id作为聚簇索引。

8.7 主键索引与辅助索引

  • MyISAM引擎

    在MyISAM引擎中，主键索引与辅助索引没有区别，叶子节点保存的都是数据记录的地址。

  • InnoDB引擎

    在InnoDB中，数据文件是主键索引按照B+Tree结构组织的，主键索引叶子节点保存的是完整的数据记录，而辅助索引叶子节点保存的是主键值。

8.8 回表与覆盖索引

在InnoDB中，辅助索引叶子节点保存的是主键的值，这代表着通过辅助索引查询时不能获取所需的所有列数据，就会按照得到的主键值再走一遍主键索引，这就是回表。

只需要在一颗索引树上就能获取所需的所有列数据，就无需回表。一般通过将被查询的字段添加到联合索引中去来实现覆盖索引。

8.9 联合索引与最左匹配原则

联合索引是指有多个索引列的索引，如(col1, col2, col3)，联合索引也是一颗B+Tree，只不过它的data域存储的是联合索引所在行的主键值，索引列会全部出现在索引树上。对于联合索引，存储引擎会对第一个索引列col1排序，如果第一列相等，则会根据第二列排序，以此类推。

使用联合索引需要遵循最左匹配原则。

创建的(col1, col2, col3)联合索引，相当于创建了(col1)、(col1, col2)、(col1, col2, col3)三个索引，因为非叶子节点是按照从左到右的顺序进行排序，所以当遇到第一个非等值查询时联合索引就会失效。

其中，等值查询时顺序可调整，MySQL优化器会自动优化为col1、col2、col3的顺序。

9. 索引失效原因

可以通过explain判断SQL语句的执行情况。

  1. 查询条件使用不等式；
  2. 查询条件类型不一致，如查询类型是字符串，传入的值为数字类型；
  3. 查询条件使用函数和数学运算；
  4. 使用前缀%模糊查询
  5. 不遵循最左匹配原则
  6. 在where子句中对字段进行null值判断
  7. MySQL优化器认为不走索引效率更高，可通过force index强制走索引

10. 主键为什么要整形、自增&&主键为什么不用UUID

  1. UUID是字符串，比整形消耗更多的空间
  2. 在B+树中进行查找时需要比较大小，整形比较运算比字符串更快
  3. 自增的主键索引在磁盘中会连续存储，提升了页的填充率，并且进行范围查询时效率很高
  4. 新增数据时会在叶子结点的末尾添加，避免了重新定位和寻址
  5. 减少页分裂和碎片的产生。因为UUID是随机的，写入乱序，InnoDB不得不做频繁地页分裂来为新行分配空间，导致移动大量的数据；并且由于频繁地页分裂，页会变得稀疏且不规则，产生大量碎片。
  6. 因为写入随机，可能写入的页已经不在内存中，需要从磁盘中读取目标页到内存，导致大量随机IO。

自增主键的缺点：

  1. 通过主键自增的特性，可以获取业务的增长信息
  2. 高并发下，因为所有的插入位置都在一起，按主键插入会导致明显的锁竞争。

11. 建立索引的原则

  1. 选择唯一性索引。
  2. 为经常需要排序、分组和联合操作的字段添加索引
  3. 为常用作查询条件的字段添加索引
  4. 限制索引数量
  5. 尽量使用前缀来索引
  6. 删除不再使用或很少使用的索引
  7. 最左前缀匹配原则
  8. 选择区分度高的列作为索引
  9. 尽量利用联合索引扩展索引，而不是新建索引

12. 为什么非主键索引叶子节点存储主键值

  • 节省内存空间

    主键索引与辅助索引各自维护各自的B+Tree结构，数据只存储一份，辅助索引通过获取主键值再去主键索引的B+树上找到对应的数据，节省了空间。

  • 保证数据一致性

    如果非主键索引叶子节点也保存一份数据，那么插入、更新、删除数据时会带来数据一致性问题。

13. 常见优化手段

  1. 查询时尽量不要使用SELECT *，而是只返回需要的字段；

    节省资源、网络开销；SELECT *很可能用不到覆盖索引，造成回表

  2. 只需要一条记录时，使用limit 1；

    加上limit 1后，找到对应的一条记录后就不会继续向下扫描，大大提高了效率；

  3. 尽量避免在where子句中使用or连接条件

    使用or可能会造成全表扫描：如果or后面字段没有索引，前面字段有索引，需要三步：全表扫描+索引扫描+合并，不如直接全表扫描。

  4. 优化limit分页

    偏移量越大，效率越低。因为MySQL是先获得偏移量+要取的条数的数据，再把前面偏移量数据去除后才返回；

    建议使用order by+索引或者通过where跳过偏移量；

  5. 避免使用前缀%进行模糊查询

    前缀%模糊查询会使索引失效

  6. 使用where语句限制要查询数据，避免返回多余的行

    需要什么数据就返回什么数据

  7. 避免在索引列上进行数学和函数运算

    会使索引失效

  8. Inner Join、Left join、right join中尽量使用Inner join，如果是left join，左边表结果尽量小

    • Inner join内连接：两张表进行连接查询时，保留完全匹配的结果集
    • left join左连接：两张表进行连接查询时，返回左表中所有的行和完全匹配的结果集
    • right join：两张表进行连接查询时，返回右表中所有的行和完全匹配的结果集

  9. 避免在where子句中使用不等于和大于小于操作符

    会使索引失效

  10. 遵循最左匹配原则

    不遵循最左匹配原则，索引会失效

  11. 考虑在常用where查询或order by的列上建立索引

  12. 插入数据过多，考虑批量操作

  13. 使用覆盖索引避免回表

  14. 慎用DISTINCT

    distinct会过滤重复记录，在字段很多的时候会大大降低查询效率

  15. 删除冗余和重复索引

    冗余和重复索引会加大维护成本

  16. 避免大规模删除修改数据

    大规模删除、修改数据会造成CPU占用较高，影响别人对数据库的访问

  17. where中考虑使用默认值代替null

    null判断很可能会使索引失效

  18. 避免大量表连接

  19. in与exists合理使用

    in相当于先执行子查询，在进行外围查询的验证，适合子查询数据较少的情景；

    exists相当于先执行主查询，获得数据后在放到子查询中条件验证，根据结果true或false判断结果是否保留，适合外围查询数据较少的情景。

  20. 尽量用union all替换union

  21. 避免建立过多索引

  22. 尽量使用数字类型

  23. 建立索引要选择区分度高的字段

  24. 避免向客户端返回过多数据

  25. 尽可能使用varchar替代char

  26. 在执行group by前过滤多余数据

  27. 如果字段是字符串，在where中一定要用引号，否则索引失效

  28. 善用explain分析SQL语句

14. 一条SQL执行很慢的原因

一条SQL执行很慢需要分为两种情况，偶尔执行很慢、一直执行很慢。

  • 偶尔执行很慢

    偶尔执行很慢说明不是SQL语句本身的问题。

    1. 数据库在刷新脏页

      往数据库中添加、更新一条数据后，会先在内存中更新，并不会立即持久化到硬盘中，而是写到redo log中，等到空闲时再刷新到硬盘中去。

    2. 发生了锁竞争

      别的事务加锁了，只能等待锁的释放。

      可以通过show processlist查看当前状态。

  • 一直执行很慢

    1. 索引失效

      可能是字段上没有索引或是索引失效

    2. 数据库优化器选错了索引

      执行SQL前数据库优化器会判断是否走索引以及走哪个索引效率更高，优化器通过索引的区分度进行判断。又因为优化器不会通过遍历来判断区分度，而是通过采样的方式来预测索引的区分度，所以可能出现错判。

      可以通过show index from table_name来判断索引的基数与实际是否符合，如果不符合，通过analyze table table_name来重新统计索引基数。

    可以通过force index(index_name)来强制使用索引。

15. 主从复制

MySQL主从复制主要涉及三个线程：binlog线程、IO线程和SQL线程。

  • bin log线程：负责将主服务器上的数据更改写入bin log中。
  • IO线程：负责从主服务器上读取bin log，并写入从服务器的中继日志relay log。
  • SQL线程：负责读取中继日志，解析出主服务器上已经执行的数据更改并在从服务器上重放(relay)。

MySQL主从复制主要用于读写分离，主服务器用于处理写操作和实时性要求高的读操作，从服务器处理读操作。

主从复制可以分为异步复制和半同步复制：

  • 异步复制

    主库写binlog成功之后就返回客户端结果，不会确认从库是否收到。

    主库写bin log并返回客户端结果后崩溃了，从库没有收到bin log日志事件。会造成数据丢失。

  • 半同步复制

    MySQL在5.7后为了解决数据丢失问题，新增了AFTER_SYNC的半同步复制方式，执行流程如下：

    1. 主库将事务写入binlog；
    2. 通知从库写relaylog；
    3. 主库等待至少N个从库返回已写入relaylog的回复，主库提交事务；
    4. 主库将操作结果返回给客户端。

    半同步复制可以保证在返回客户端前，其他事务看不到当前事务做出的更改，并且会确保bin log会写入slave的relay log。

MySQL主从架构可以大大提高系统并发性能：

  • 主从服务器分别负责写和读，极大缓解了锁的竞争
  • 从服务器可以使用MyISAM，提高查询性能，节约系统开销
  • 增加冗余，提高灵活性。

读写分离一般通过代理模式实现，代理服务器根据应用层的读写请求决定转发到哪台服务器。

16. explain关键字

explain可以展示一条SQL的执行计划，主要关键字有type、key、rows。

  • type

    连接类型，表示数据库引擎查询表的一种方式。常见的有6种方式：all、key、range、ref、eq_ref、const，从左到右，效率增强。

    • all：全表扫描
    • index：按照索引顺序全表扫描
    • ref：查找条件使用了索引但不为主键和unique，虽然使用了索引但索引列的值不唯一。
    • eq_ref：使用了主键或者唯一性索引
    • const：使用主键作为查询条件时，优化器能把这次查询优化为一个常量。

  • key

    查询用到的索引

  • rows

    执行计划在预估的扫描行数，非精确值，值越小，效率越高。

17. 慢查询排查

  1. 开启慢查询日志，设置slow_query_log参数为ON，查看慢查询show status like 'slow_query_log';
  2. show processlist查询当前查询状态；
  3. explain查看慢查询SQL语句。

18. 数据库范式

  • 第一范式

    列具有原子性，属性不可分

  • 第二范式

    每个非主属性完全依赖于键码

  • 第三范式

    非主属性不传递函数依赖于键码，即不存在传递依赖。

19. MySQL数据一致性

MySQL数据一致性包括单机数据一致性和集群数据一致性。

单机数据一致性

MySQL保证单机数据一致性主要是通过MySQL的bin log和InnoDB的redo log。

  • bin log

    bin log是MySQL Server层维护的二进制日志，记录了所有对数据库的更改操作。bin log主要有两个作用：数据恢复和数据复制。

    MySQL崩溃后，通过bin log恢复写入bin log但没有写入磁盘文件的数据。

    数据复制则是指主从复制时从节点读取主节点的bin log到自己的本地relay log中进行重演。

  • redo log

    redo log是InnoDB用来实现事务持久性的机制。包括两个部分redo log file和redo log buffer。

    redo log file保证数据落盘，是持久的；redo log buffer在内存中，是非持久的，会批量刷新到磁盘，提高吞吐量。

为了保证数据一致性，写操作先redo log预提交，再写bin log，最后redo log提交。

数据以bin log为准，Crash Recovery时有以下情景：

  1. binlog有记录，redolog状态commit：正常完成的事务，不需要恢复；
  2. binlog有记录，redolog状态prepare：在binlog写完提交事务之前的crash，恢复操作：提交事务；
  3. binlog无记录，redolog状态prepare：在binlog写完之前的crash，恢复操作：回滚事务；
  4. binlog无记录，redolog无记录：在redolog写之前crash，恢复操作：回滚事务；

集群下的数据一致性

集群下的数据一致性就是要保证主从复制的数据一致性。

主从复制可以分为异步复制和半同步复制。

异步复制会造成数据丢失，半同步复制的AFTER_SYNC可以保证数据一致性。

  1. 主库将事务写入binlog；
  2. 通知从库写relaylog；
  3. 主库等待至少N个从库返回已写入relaylog的回复，主库提交事务；
  4. 主库将操作结果返回给客户端。

半同步复制可以保证在返回客户端前，其他事务看不到当前事务做出的更改，并且会确保bin log会写入slave的relay log。
